# Phase 9 — UI Design Contract

> Visual and interaction contract for the persistence + reconnect surfaces.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Stack note:** Vector is a native macOS app (Rust + wgpu/Metal + AppKit via `objc2-app-kit`).
> No web framework, no shadcn, no Tailwind. The "components" in this contract map to wgpu
> render pipelines (`ChromeQuadPipeline`, `CellPipeline` glyph atlas) and AppKit constructs
> (`NSWindow` tab title strings). The full design system was established in Phases 3–5 and
> codified in `crates/vector-theme::ChromePalette` — Phase 9 reuses those tokens verbatim
> and introduces ZERO new color or typography tokens.

### Phase-specific geometry (locked, do not deviate)

| Constant | Value | Rationale / Source |
|----------|-------|--------------------|
| **Status bar height** | **24 px** | Slightly taller than the 22 px Phase 6 picker row; one glyph cell (JetBrains Mono 14pt ≈ 18 px line) + 4 px vertical inset top + bottom. 4 px multiple. Closest reference: Phase 5 `SEARCH_BAR_HEIGHT_PX = 32`; Reconnect is more compact because it overlays a live pane rather than replacing chrome. CONTEXT.md D-07 says "~22 px"; rounded UP to the nearest 4-multiple for spacing-scale compliance. |
| **Status bar width** | **= pane content width** | Full-width within the pane viewport (NOT window-wide). Anchored to the pane's compositor rect, NOT the window. Multi-pane (split-view) windows show one status bar per reconnecting pane. |
| **Status bar vertical anchor** | **Top edge of pane viewport** (y = pane_rect.y) | Overlays the first row of the live grid. The underlying grid row is NOT scrolled out — the bar simply paints over it for the duration of the reconnect. CONTEXT.md D-02: grid + scrollback remain fully visible underneath (visible 22-23 rows out of 24, depending on pane size). |
| **Glyph row baseline within bar** | top + 4 px inset; bottom + 4 px inset | Centers a 14pt JetBrains Mono row in the 24 px bar (16 px ascent box). |
| **Z-order / composition order** | After per-pane compositor (terminal grid + selection + cursor + border); before window-level chrome (TintStripePipeline, SearchBarPass, ToastPass, PickerPass) | RESEARCH.md §Pattern 4 composition order spec. Hook site: `crates/vector-app/src/app.rs` line ~1055. |

### Exceptions

- **24 px bar height** is the only deviation justification: pure-multiple-of-4 alternatives were 20 px (clips JetBrains Mono 14pt descenders at the 4 px bottom inset; cell metrics from `vector-render/src/atlas.rs` JetBrains Mono `cell_height ≈ 18 px` at 14pt @1×) or 28 px (over-tall; introduces visible top-of-pane intrusion when grid row #0 has live content). 24 px is the minimum 4-multiple that fits a 14pt mono cell + symmetric 4 px insets. Approved.

All other spacing values are strict multiples of 4.

## Typography

Status bar text renders through the existing `CellPipeline` glyph atlas (Phase 3 Plan 03-02). Vector ships JetBrains Mono bundled for snapshot-test determinism (Plan 03-02). Status bar reuses this exact font — NO separate UI font is introduced.

| Role | Size | Weight | Font | Usage |
|------|------|--------|------|-------|
| Body (status bar text) | **14pt** (=18 px @1×) | regular | JetBrains Mono (bundled) | The full message: `Reconnecting to {profile}… (attempt N)` |
| Label (tab title `[reconnecting]` badge) | system tab title size | regular | system font (SF) — AppKit picks | The badge suffix appended by `format_tab_title` |

**Rules:**

- Exactly **2 sizes** declared (1 chrome size + 1 system-font size for the tab badge). No headings, no display sizes, no captions.
- Exactly **1 weight** (regular) for the status bar; AppKit decides for the tab badge.
- Status bar text size **matches the terminal grid font size** (whatever the user has configured in their `~/.config/vector/config.toml` `[font]` section). If the user has configured 16pt mono grid cells, the status bar text scales to 16pt. **Status bar text always tracks the active pane's grid font size**, never a fixed value. This is the one place Vector deviates from "pin a typography size in pixels" — it's a deliberate property of an overlay anchored INSIDE a terminal pane.
- Line height: cell-pipeline default (1.0 of cell height; no extra leading). Single-line bar — no wrapping.
- **No ligatures inside the status bar.** Even if the user has `ligatures = true` in their config, the status bar's ASCII-only copy ("Reconnecting", "attempt") will not produce ligatures naturally — but explicitly disable in case the JetBrains Mono set ever ships a `tt`/`mm` ligature.

## Color

Vector inherits its color contract from Phase 5 (`crates/vector-theme/src/builtins.rs`). Phase 9 introduces **ZERO new color tokens** — it composes status bar visuals from existing `ChromePalette` fields.

| Role | Token (existing `ChromePalette` field) | Value (Dark) | Value (Light) | Usage in Phase 9 |
|------|-----------------------------------------|--------------|----------------|-----------------|
| Dominant (60%) — status bar background | `chrome.surface` | `Rgba(0x1c1c1e, α=0xe6)` | `Rgba(0xf4f4f5, α=0xe6)` | The 24 px bar fill (90% opaque over the live grid below) |
| Secondary (30%) — status bar primary text | `fg` (palette grid fg, NOT `fg_muted`) | `Rgb(0xffffff)` | `Rgb(0x1d1d1f)` | The `Reconnecting to {profile}…` text portion |
| Accent (10%) — attempt counter | `chrome.warning` | `Rgb(0xffd60a)` (amber) | `Rgb(0xff9500)` (orange) | **Reserved exclusively for the `(attempt N)` substring**, signaling "transient anomaly, not an error" |
| Secondary text — profile label inside text | `chrome.fg_muted` | `Rgb(0x8e8e93)` | `Rgb(0x8e8e93)` | The `{profile}` substring rendered slightly muted to de-emphasize against the action verb "Reconnecting" |
| Dimmed cursor (S2) | `cursor` × alpha 0.4 | `Rgb(0xc9d1d9)` @ α 0.4 | `Rgb(0x5a5a5f)` @ α 0.4 | Cursor in a reconnecting pane: rendered with 40% of normal alpha; stops blinking (CONTEXT.md D-02) |

### Accent reserved for

The amber/orange `chrome.warning` token in this phase is reserved **exclusively** for:

1. The `(attempt N)` substring of the inline status bar.

Nothing else in this phase uses warning color. Specifically:
- The bar background does NOT use warning (uses neutral `chrome.surface`).
- The "Input ignored — reconnecting" toast (S3) uses the **existing Phase 5 Info toast** appearance — NOT warning. We are not escalating a transient locked-input event into a warning-colored alert.
- The tab `[reconnecting]` badge does NOT use color — it's a plain text suffix in the system tab title bar (AppKit decides).

### Destructive color

**Not used in this phase.** Phase 9 has no destructive actions. Closing a reconnecting pane with Cmd-W is the existing close-pane affordance and uses no new confirmation UI (existing behavior: pane closes immediately, the reconnect loop's cancellation token aborts the backoff sleep).

### Contrast verification (WCAG AA, 4.5:1 minimum)

| Foreground / Background | Dark contrast | Light contrast | Pass AA (≥ 4.5:1)? |
|--------------------------|---------------|----------------|---------------------|
| `fg #ffffff` on `chrome.surface #1c1c1e @ α=0.9` over `bg #0d1117` | ≈ **15.8:1** | n/a | ✅ |
| `fg #1d1d1f` on `chrome.surface #f4f4f5 @ α=0.9` over `bg #ffffff` | n/a | ≈ **15.5:1** | ✅ |
| `chrome.warning #ffd60a` (amber) on `chrome.surface #1c1c1e` (dark) | ≈ **12.4:1** | n/a | ✅ |
| `chrome.warning #ff9500` (orange) on `chrome.surface #f4f4f5` (light) | n/a | ≈ **3.7:1** | ⚠️ **FAIL for body text; PASS for "large text" (≥ 18pt regular).** Status bar text IS 14pt — so this is **conditional**. See mitigation below. |
| `chrome.fg_muted #8e8e93` on `chrome.surface #1c1c1e` (dark) | ≈ **4.7:1** | n/a | ✅ |
| `chrome.fg_muted #8e8e93` on `chrome.surface #f4f4f5` (light) | n/a | ≈ **4.5:1** | ✅ (exactly at floor) |

**Light-mode amber mitigation (required):** The `(attempt N)` text in light mode must be drawn at **bold weight** (existing JetBrains Mono Bold glyph in atlas — same atlas already loads it for terminal bold-attribute cells in Phase 3) to satisfy WCAG AA "large/bold text" relaxed threshold (3:1). This bumps the effective contrast to the bold-text exception and is the **only** in-phase typography weight deviation. Dark-mode amber needs no such treatment (12.4:1 is far above AA).

**Decision:** `(attempt N)` substring renders **bold in light mode, regular in dark mode**. Implementation note for the planner: the existing `CellPipeline` already supports per-glyph bold via the SGR-bold attribute path — reuse that, do not introduce a new shader uniform.

### High-contrast mode

macOS "Increase contrast" accessibility setting is honored automatically by NSColor system tokens — but Vector uses `ChromePalette` fixed hex values (not `NSColor::labelColor`), so high-contrast mode is NOT auto-applied to the status bar. **For v1, this is accepted as-is.** Backlog item: "Phase 9 status bar honors macOS Increase Contrast" (deferred).

### Screen-reader / accessibility tree

**Not applicable for v1.** wgpu surfaces do not expose to the macOS Accessibility tree, and the terminal grid itself is not AX-readable in v1 (this is a v2 concern — see `TERM-V2-*` family). The reconnect status is announced indirectly via:
- The tab title's `[reconnecting]` badge (system-readable via VoiceOver's window-title path).
- `tracing` logs (for diagnostics, not user-facing).

No AX hooks are required of Phase 9. Document as a known v1 limitation.

## Animation / Motion

Vector's existing motion contract (Phase 5 `ToastPass`): 120 ms fade-in, 200 ms fade-out, instant under macOS Reduce Motion. Phase 9 reuses these durations to keep motion identity consistent across overlays.

| Event | Animation | Duration | Reduce Motion behavior |
|-------|-----------|----------|------------------------|
| Status bar appears | Fade-in (alpha 0 → 1, linear) | **120 ms** (reuse `TOAST_FADE_IN_MS`) | Instant appear |
| Status bar disappears (on reconnect success) | Fade-out (alpha 1 → 0, linear) | **200 ms** (reuse `TOAST_FADE_OUT_MS`) | Instant disappear |
| Attempt counter increments | **No animation.** Value change is a glyph re-rasterization on the next frame. | n/a | n/a |
| Spinner / dots / progress bar | **None.** The bar text is static between attempts. The presence of the bar IS the activity indicator. | n/a | n/a |
| Tab badge transition `[remote]` → `[reconnecting]` | None — AppKit re-sets the window title synchronously | n/a | n/a |
| Dimmed-cursor transition (Active → Reconnecting) | Instant alpha-multiplier change (1.0 → 0.4) on next frame; cursor blink halts simultaneously | n/a | n/a |
| Cursor blink during Reconnecting | **Halted.** Cursor is statically rendered at 40% alpha. CONTEXT.md D-02. | n/a | n/a |
| Dropped-keystroke flash | None at the keystroke level. ONLY the one-shot Info toast (S3, first keystroke). Subsequent keystrokes during the same Reconnecting span produce **zero visual or audible feedback** (deliberate — flooding flashes on every keystroke would be worse UX). | n/a | n/a |

### Debounce — DO NOT SHOW BAR FOR SUB-250 ms BLIPS

Per CONTEXT.md "Discretion" → researcher decision: **debounce the bar at 250 ms.**

- On transport death detected, start a 250 ms timer before showing the bar.
- If the new transport is established within those 250 ms (rare but possible — e.g. immediate retry succeeded), do NOT flash the bar at all.
- If the timer elapses, show the bar with fade-in.

Rationale: A 120 ms fade-in followed by an instant disappear-on-success creates a "blink" the user perceives as a glitch, not as progress. The 250 ms debounce is below the threshold of perceptible delay (~300 ms) for noticing a network event, but above the threshold for "did something flicker?".

Implementation note: the debounce is per-pane and resets on every `Active → Reconnecting` transition.

### Frame rate / repaint rule

The status bar lives in the per-pane render path. Phase 3's render-on-dirty discipline applies: a pane in the Reconnecting state has its repaint scheduled by:
1. Attempt-counter increments (once per backoff slot, ≤ once per 30s).
2. Fade-in / fade-out frame requests during the 120 ms / 200 ms transitions.

There is **no continuous repaint**. The bar is static between attempts; idle CPU during a long reconnect must stay below the Phase 3 RENDER-03 budget (< 1% on Apple Silicon).

## Input-Lock Affordances

CONTEXT.md D-03: input is LOCKED during the blip, not queued. Phase 9 contract for how that lock is signaled:

| Signal | Treatment | Visibility |
|--------|-----------|------------|
| **Status bar present** | Primary signal. Visible at the top of the affected pane. | Always (after 250 ms debounce). |
| **Dimmed cursor (40% alpha, no blink)** | Secondary signal. The user's eye is naturally on the cursor position; dimming it telegraphs "this terminal is asleep." | Always during Reconnecting. |
| **Cursor shape preserved** | The cursor's DECSCUSR shape (block / underline / bar) from the live grid is preserved — only its alpha is dimmed and its blink is halted. We do NOT force-revert to block cursor. | Always. |
| **First-keystroke toast** | "Input ignored — reconnecting" Info toast (S3). Once per Reconnecting span. | First keystroke only. |
| **Subsequent keystrokes during the same span** | Silent drop. No flash, no sound, no toast. | None. |
| **Audible feedback** | **None.** Vector does not ring the system bell on dropped input. Adding `NSBeep()` would be jarring during a frequent (per-blip) event. | None. |
| **Visual flash per keystroke** | **None.** Considered a "brief flash" (CONTEXT.md "Discretion") — explicitly rejected because the status bar + dimmed cursor are sufficient passive indicators. Flashing on each keystroke would be epileptic-trigger territory and is undesired. | None. |
| **Input lock applies to** | The pane in Reconnecting state ONLY. Other panes (local, or another connected DevTunnel) accept input normally. | Per-pane. |
| **Mouse selection / scrollback scrolling** | **NOT locked.** D-02: user can still scroll the scrollback while reconnecting. Text selection (Cmd-C copy) also works. Only **transport-bound input** (keystrokes → bytes sent over the wire) is locked. | Always available. |
| **Cmd-W close** | Always accepted. Closes the pane, cancels the reconnect loop via `CancellationToken`. | Always. |
| **Window-level shortcuts** (Cmd-T new tab, Cmd-D split) | Always accepted. They don't target the locked pane's transport. | Always. |

## Theme Integration

Vector's TOML config (`~/.config/vector/config.toml`) defines `[theme]` sections (Phase 5 POLISH-03). Phase 9 introduces **ZERO new theme keys** — all colors flow from existing `ChromePalette` fields documented in §Color.

### What the user can override

Nothing Phase 9-specific. The status bar's appearance changes when the user:
- Switches between `theme = "vector_dark"` and `theme = "vector_light"` (both built-in).
- Imports a `.itermcolors` file — but per `crates/vector-theme/src/palette.rs:46-47`, `.itermcolors` **only overrides grid colors, never chrome**. So a custom iTerm scheme will NOT change the status bar appearance. This is intentional (Phase 5 D-83 invariant: chrome stays consistent across grid themes).

### What the user CANNOT override (and that's a feature, not a bug)

- Status bar height (24 px hard-coded; not configurable in v1).
- Status bar position (pane-top hard-coded).
- The 250 ms debounce (not configurable in v1).
- The backoff schedule (CONTEXT.md D-08: 1/2/4/8/16/30, hard-coded; per-profile override deferred to v1.x).
- Whether the bar appears at all (cannot be disabled; disabling would defeat the safety property the bar provides).

### Future config keys (NOT in v1, document for the planner so they're not surprised)

When/if v1.x adds per-profile override, expected keys (illustrative, NOT implemented in this phase):
- `[profiles.my-box.reconnect] backoff_schedule_secs = [1, 2, 4, 8, 16, 30]`
- `[profiles.my-box.reconnect] debounce_ms = 250`
- `[profiles.my-box.reconnect] show_overlay = true`

These are listed for forward-compatibility awareness. Phase 9 MUST NOT add these keys.

## Edge Cases

| Case | Behavior |
|------|----------|
| **Pane is very narrow (< 40 cells wide)** | Truncate profile name in middle: `Reconnecting to {first8}…{last8} (attempt N)`. |
| **Pane is extremely narrow (< 28 cells)** | Replace profile name with ellipsis: `Reconnecting… (attempt N)`. |
| **Pane is impossibly narrow (< 18 cells)** | Hide status bar entirely. The tab `[reconnecting]` badge still communicates state. Repaint check uses the cell-width at frame time, not at Reconnecting-entry time — so a resize that grows the pane back above 18 cells re-shows the bar. |
| **Pane is split (multi-pane window)** | Each Reconnecting pane shows its own status bar inside its own viewport. Bars do NOT coordinate across panes; they may overlap split borders only at the pane's own top edge (never across into a neighbor's region). The compositor draws each pane's bar in its own pass. |
| **Multiple panes reconnecting simultaneously** | Each shows its own bar independently. Each has its own attempt counter (they may diverge if connections come back at different times). No global "N panes reconnecting" indicator (deferred to v1.x). |
| **Local pane (LocalDomain)** | Status bar NEVER appears. `LocalDomain::reconnect_one_shot` returns `Ok(None)` → actor exits cleanly → no Reconnecting state ever entered. CONTEXT.md "Out of scope" confirms local panes still die silently. |
| **Pane in fullscreen mode** | Bar still appears at pane-top inside the fullscreen pane. No special treatment. |
| **Pane scrolled into scrollback** | Bar still anchored to viewport top (above the scrolled grid). User can scroll while the bar is visible. CONTEXT.md D-02. |
| **Reconnect succeeds during fade-in (uncommon but possible)** | Cancel the fade-in; jump straight to fade-out from the current alpha. Avoids the "bar appears then immediately disappears" double-blink. |
| **User resizes window during reconnect** | Bar re-anchors to the new pane rect on the next frame. Bar width updates. Text re-truncates if width drops below thresholds. (CONTEXT.md "Discretion" for SIGWINCH: stored dims are sent to the new transport on swap — that's transport, not UI; the UI re-anchors purely on the next compositor pass.) |
| **User changes theme during reconnect** | Hot-reload (POLISH-01) re-renders the bar with the new `ChromePalette` on the next frame. No special handling. |
| **Attempt counter wraps past 2^32** | Will not happen in any human-realistic timeline (2^32 × 30s ≈ 4000 years). `attempt: u32` with `saturating_add` is fine. |
| **Status bar pane disappears (Cmd-W during reconnect)** | Bar removed atomically with pane. CancellationToken aborts the loop. No "ghost" bar. |
