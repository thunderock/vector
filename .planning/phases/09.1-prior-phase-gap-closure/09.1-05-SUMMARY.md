# Phase 09.1 Plan 05: Bundled App Icon (Gap E) Summary

**Authored typographic V icon SVG + wired `cargo xtask icon` subcommand + set `CFBundleIconFile=icon` so the bundled Vector.app ships the V mark on dock/Cmd-Tab/Finder.**

## Performance

- **Duration:** ~5 minutes (two task commits).
- **Task 1 commit:** `7a24131` (feat — icon.svg redesign + xtask Cmd::Icon wiring)
- **Task 2 commit:** `8ccc0aa` (feat — CFBundleIconFile=icon in Info.plist.partial; commit also captured parallel-agent vector-codespaces deletions from 09.1-02 — see Deviations below)

## What Shipped

### Task 1 — Author icon.svg + run `cargo xtask icon`

Replaced the speculative chevron design in `crates/vector-app/resources/icon.svg` with the D-19 visual spec: a 1024×1024 dark rounded-square (`#1a1a1a`, rx=184 ≈ 18% corner radius matching macOS Big Sur+ icon grid) with a centered monospaced V glyph (`#f5f5f5`, font-size 768pt, weight 700) using JetBrains Mono → Menlo fallback.

Wired `Cmd::Icon` in `xtask/src/main.rs` so the pre-existing `xtask::icon::generate_icns` function is reachable as `cargo xtask icon`. Ran it: produced `crates/vector-app/resources/icon.icns` (100,997 bytes, `Mac OS X icon, ic12 type`) containing all 10 standard iconset sizes (16×1 through 512×2) via `rsvg-convert` + `iconutil`.

### Task 2 — Wire CFBundleIconFile

Added explicit `<key>CFBundleIconFile</key><string>icon</string>` to `crates/vector-app/resources/Info.plist.partial`. cargo-bundle 0.10 also auto-injects this from `[package.metadata.bundle].icon`, but the partial-plist entry makes the contract explicit and survives future cargo-bundle behavior changes.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] Wired missing `Cmd::Icon` xtask subcommand**
- **Found during:** Task 1, when `cargo xtask icon` failed with `error: unrecognized subcommand 'icon'`.
- **Issue:** `xtask/src/icon.rs` existed with the full `rsvg-convert` + `iconutil` pipeline (authored Phase 1, Plan 01-04), and was already declared as `mod icon;` in `main.rs`, but no `Cmd::Icon` variant or match arm routed CLI invocations to it. The function was only reachable via `xtask::dmg::finalize` calling `super::icon::generate_icns(sh)` directly.
- **Fix:** Added `Icon` to the `Cmd` enum (with doc string `/// Regenerate crates/vector-app/resources/icon.icns from icon.svg.`) and added a match arm `Cmd::Icon => icon::generate_icns(&sh)`.
- **Files modified:** `xtask/src/main.rs`
- **Commit:** `7a24131`

### Other Deviations

**2. [Process — parallel-agent race] Task 2 commit picked up unrelated `vector-codespaces` deletions**

- **Context:** This plan executed in parallel with 09.1-02 (Phase 7 Codespaces UI cleanup, Gap B). Between my `git add crates/vector-app/resources/Info.plist.partial` and `git commit`, the 09.1-02 agent appears to have staged the deletion of the entire `crates/vector-codespaces/` tree (Cargo.toml, src/, tests/, fixtures — 17 files, 1735 lines deleted).
- **Outcome:** My commit `8ccc0aa` recorded "feat(09.1-05): set CFBundleIconFile=icon in Info.plist.partial" but also captured the vector-codespaces deletions. The commit message documents only the icon work; the deletions are unrelated.
- **Impact:** Net repo state is correct (vector-codespaces is removed per Gap B intent; icon plist is set per Gap E intent). The commit boundary is fuzzy — a code archaeologist tracing Gap B will need to look at commit `8ccc0aa` despite its Gap E subject line.
- **Mitigation:** 09.1-02's SUMMARY should reference both `8ccc0aa` (codespaces dir + Cargo.toml/app.rs/lib.rs deletions if those land in their own commit, or the dir deletions here + their other changes elsewhere) so the cross-link survives.

**3. [Scope] `icon.icns` not committed to git despite plan frontmatter listing it**

- **Context:** Plan frontmatter at line 7-11 lists `crates/vector-app/resources/icon.icns` under `files_modified`, but `.gitignore` lines 5-6 exclude `icon.iconset/` and `icon.icns`.
- **Decision:** Honored the gitignore. icon.icns is a build artifact regenerated by `xtask::dmg::finalize` (which calls `icon::generate_icns` before `cargo bundle`) on every DMG build. Committing it would create needless diff churn on every regeneration (rsvg-convert + iconutil output is not byte-deterministic across librsvg versions).
- **Source of truth:** `icon.svg` (committed). Regeneration step: `cargo xtask icon` (now reachable as of commit `7a24131`).
