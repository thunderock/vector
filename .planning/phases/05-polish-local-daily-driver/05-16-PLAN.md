Four wgpu pipelines exist, compile, and pass geometry unit tests, but none are instantiated in `RenderHost` and none are invoked from the live render loop. UI-SPEC §11 specifies the exact draw order:
1. Per-pane compositor (grid + Phase-4 active-pane border) — already wired (app.rs:300–380).
2. Tint stripe (window chrome, above all panes).
3. OSC-8 hover underline (lives inside grid — already part of grid attributes).
4. Search bar (per-pane, inside active pane border).
5. Toast (window chrome, top of content).
6. Profile picker scrim + modal (modal, dims everything).

This plan:

1. **RenderHost owns the four chrome pipelines** (mutable, one instance per surface so they share the surface format). Construct them in `RenderHost::new` right after the wgpu surface is up.

2. **App::render_window invokes the chrome draws AFTER the per-pane compositor loop** but BEFORE `frame.present()`. The acquired `wgpu::SurfaceTexture` + `TextureView` from the existing acquire_frame is reused; a new `wgpu::CommandEncoder` is created for the chrome passes (or the existing encoder is held across pane render — depends on the Compositor API; investigate). For each chrome surface, conditionally call its `update_*` + `draw` methods inside a single chrome `wgpu::RenderPass` with LoadOp::Load (so we composite over the grid).

3. **Wire the orchestration to App state**:
   - Tint: read `app.profile_picker.entries[active].tint` (or via `current_config.profile[active_profile_name].tint`); if Some, `tint.set_color` + `tint.update_quad` + `tint.draw`. If None, skip (UI-SPEC §9.3: stripe pass entirely skipped).
   - SearchBar: `if app.search_bar.open { search_bar_pass.update_for_pane(...); search_bar_pass.draw(); }`. Coordinates come from the active pane's rect (which we computed inside the existing render_window loop).
   - Toast: `if let Some(toast) = app.toasts.current() { toast_pass.update(...); toast_pass.draw(); }`. Use `alpha_at(elapsed, total, reduce_motion=false)` for the alpha; the existing ToastBanner.shown_at gives `elapsed`.
   - Picker: `if app.profile_picker.open { picker_pass.draw_scrim(...); picker_pass.draw_modal(...); }`. Use `picker_layout(...)` to compute the rect.

4. **Tick the toast stack each frame** so info toasts auto-dismiss after 5s: call `app.toasts.tick(Instant::now())` at the top of `render_window`.

5. **Glyph rendering for chrome text** is OUT OF SCOPE for this plan. The four passes draw their backgrounds and active-match border; text rendering (query string, counter, profile names, toast text) reuses the Phase-3 glyph atlas via the existing CellPipeline, but the orchestration of glyph runs over chrome rects is a separate concern owned by a future task or by extending the existing chrome_quad pipeline. v1 acceptance: the chrome BACKGROUNDS appear; the rendered text is a Phase-5.x polish item. This is consistent with UI-SPEC §5.2 ("32 px height background bar.
