## Summary

The Dev Tunnels stack in Vector is **already provider-agnostic at every layer below the app's sign-in driver**. `AuthProvider::GitHub(String)` exists and emits `github <token>` (`model.rs:69-92`); `api.rs` sends `auth.format_header()` verbatim on every call; the agent already drives a GitHub device flow and dispatches `Authorization::Github`. The only Microsoft-specific code is (a) the app-side device-flow driver `device_flow_microsoft.rs` + `MicrosoftTokenStore`, and (b) a layer of app-side UI symbols (`UserEvent` variants, menu install fn, modal button, ObjC subclasses, footer copy) that Phase 9.1 *just* built/renamed toward Microsoft (commits 5ffa51e, 622d3fe). This phase is a near-mechanical mirror + rename, gated by one spike.

The single real unknown is **token-path acceptance**: does the Dev Tunnels Management API accept a GitHub token issued by the **Dev Tunnels GitHub App** (`Iv1.e7b89e013f801f03`), and does that GitHub App have **device flow enabled**? GitHub returns a distinct `device_flow_disabled` error if not ÔÇö the spike must check for it explicitly. A secondary unknown surfaced by the spike: whether the agent's current `gh`-CLI-issued token (`178c6fc778ccc68e1d6a`) is accepted by the relay at all ÔÇö the spike should test **both** client IDs against `GET /api/v1/tunnels` to settle D-4.

GitHub's device flow differs from Microsoft's in two ways the new driver must handle: (1) GitHub defaults to **form-encoded** responses unless `Accept: application/json` is sent (the agent already sets this header ÔÇö mirror it), and (2) a refresh token is issued **only if the GitHub App opted into "expiring user tokens"**; the access token then expires in 8 hours and the refresh token in 6 months. The token store must persist access + optional refresh + expiry (the existing `MicrosoftTokens` shape already covers this with `refresh_token: Option<String>`).

**Primary recommendation:** Build the spike first as a single `#[ignore]`-gated env-driven integration test in `crates/vector-tunnels/tests/`, proving device-flow completion **and** `list_tunnels` 200 against both client IDs. Then mirror `device_flow_microsoft.rs` Ôćĺ `device_flow_github.rs` + `GitHubTokenStore` (TDD, wiremock), then rename the app-side Microsoft surface to GitHub, then run the live smoke (resumed Phase 9 UAT Test 3+). The arch-lint needs **no** SCAN_PATHS change ÔÇö it already scans the whole `vector-tunnels/src` tree.

### Pattern 1: GitHub device-flow driver (mirror of `MicrosoftAuth`)
**What:** A `GitHubAuth` struct with `new(client_id)` + `with_endpoints(...)` test seam, `start_device_flow()`, `poll_until_authorized(...)`, and `refresh(...)`. Identical control flow to `device_flow_microsoft.rs:114-268`.
**Key differences from the Microsoft driver:**
- **Set `Accept: application/json` on BOTH POSTs.** GitHub defaults to `application/x-www-form-urlencoded` responses; without the header `serde_json::from_str` fails. The agent's `request_device_code` (`auth.rs:149`) and `poll_token` (`auth.rs:224`) already do this ÔÇö copy it. The Microsoft driver does NOT set this header (Microsoft always returns JSON), so this is the one line you must add that the mirror source lacks.
- **Endpoints:** `https://github.com/login/device/code` (device code) and `https://github.com/login/oauth/access_token` (token). No `scope` needed for the Dev Tunnels GitHub App identity flow ÔÇö but mirror the agent which passes `scope=read:user` for the `gh` CLI app; for the Dev Tunnels GitHub App the spike determines whether a scope param is required (GitHub Apps ignore OAuth scopes; permissions are set on the app). **Recommendation:** send no `scope` for the Dev Tunnels GitHub App, or an empty string; confirm in the spike.
- **Error code set** (poll endpoint) ÔÇö handle exactly these (verified against GitHub docs):
  - `authorization_pending` Ôćĺ keep polling at current interval
  - `slow_down` Ôćĺ add **5 seconds** to interval (GitHub's documented behavior; the agent does `interval += 5s` at `auth.rs:246`). Note Microsoft's driver *doubles* the interval (`device_flow_microsoft.rs:212`) ÔÇö GitHub adds 5s. Mirror the **agent's** +5s, not the Microsoft driver's doubling.
  - `expired_token` Ôćĺ `DeviceCodeExpired`
  - `access_denied` Ôćĺ `AccessDenied`
  - `device_flow_disabled` Ôćĺ **distinct fatal error** ÔÇö this is the signal the GitHub App lacks device flow. Surface it as a typed variant so the spike + UI can report it clearly.
  - `incorrect_client_credentials` / `incorrect_device_code` / `unsupported_grant_type` Ôćĺ fatal `Unexpected`
- **`verification_uri` vs `verification_url`:** GitHub returns `verification_uri`; the agent defensively checks both (`auth.rs:179-181`). Mirror that defensiveness.

**Example (the header line the mirror must add):**
```rust
// Source: crates/vector-tunnel-agent/src/auth.rs:147-152 (already correct for GitHub)
let resp = self.http
    .post(url)
    .header(reqwest::header::ACCEPT, "application/json")  // REQUIRED for GitHub
    .form(&[("client_id", client_id), ("scope", scopes)])
    .send().await?;
```

### Pattern 2: Token store (mirror of `MicrosoftTokenStore`)
**What:** `GitHubTokenStore` storing a JSON blob `{access_token, refresh_token, expires_at_unix}` under `Secrets::GITHUB_REFRESH_ACCOUNT` (constant already exists, `vector-secrets/src/lib.rs:53`).
**The `MicrosoftTokens` shape already handles GitHub's reality:** `refresh_token: Option<String>` ÔÇö `None` when the GitHub App has NOT opted into expiring tokens (classic OAuth-App-style non-expiring token), `Some` when it has. `expires_at` math: GitHub's token endpoint returns `expires_in` (8h = 28800s) only when expiring tokens are on; the `parse_token_response` `.unwrap_or(3600)` fallback (`device_flow_microsoft.rs:297`) is wrong for GitHub non-expiring tokens ÔÇö recommend a large/sentinel fallback (or treat absent `expires_in` as "never expires" and skip refresh). **Confirm in the spike which case the Dev Tunnels GitHub App is in.**
**Refresh:** GitHub's refresh uses `grant_type=refresh_token` + `refresh_token` + `client_id` against the same token endpoint, returning a NEW access token AND a NEW refresh token (rotation). The Microsoft `refresh` shape (`device_flow_microsoft.rs:232-268`) maps directly; GitHub's invalid-refresh error is `bad_refresh_token` (map to `RefreshExpired`), not Microsoft's `invalid_grant`.

### Anti-Patterns to Avoid
- **Do NOT `#[derive(Debug)]` on any struct with `access_token`/`refresh_token`/`device_code`/`user_code` fields.** Pitfall-14 arch-lint (`crates/vector-arch-tests/tests/no_token_in_debug_or_log.rs`) scans the entire `vector-tunnels/src` tree and fails the build. Write manual `Debug` exactly as `device_flow_microsoft.rs:39-81` does.
- **Do NOT `tracing::*!` any token-named local.** Same lint, second test (`no_token_in_tracing_calls`). The `tracing::info!("github_device_flow_complete")` literal-string logs in the Microsoft driver are fine; just never interpolate a token.
- **Do NOT copy Microsoft's `slow_down` doubling.** GitHub adds 5s.

## Common Pitfalls

### Pitfall 1: Missing `Accept: application/json` Ôćĺ non-JSON parse failure
**What goes wrong:** GitHub returns `access_token=gho_...&token_type=bearer&scope=` form-encoded; `serde_json::from_str` errors out as "non-JSON response."
**Why it happens:** The Microsoft driver (the mirror source) omits this header because Microsoft always returns JSON.
**How to avoid:** Set `.header(reqwest::header::ACCEPT, "application/json")` on both POSTs. The agent already does this correctly.
**Warning signs:** `Unexpected("non-JSON ... response")` errors even on a valid code.

### Pitfall 2: `device_flow_disabled` ÔÇö the GitHub App lacks device flow
**What goes wrong:** `POST /login/device/code` (or the token poll) returns `error: device_flow_disabled`; sign-in can never complete.
**Why it happens:** "Enable Device Flow" is an opt-in checkbox in the GitHub App settings. The Dev Tunnels GitHub App is Microsoft-owned; we cannot toggle it. This is the phase's headline risk.
**How to avoid:** The spike (Plan 09.2-01) tests this BEFORE any driver is built. If disabled, fall back per CONTEXT: loopback OAuth redirect, or test whether the `gh` CLI client ID's token is accepted by the relay instead.
**Warning signs:** Spike returns `device_flow_disabled`.

### Pitfall 3: Relay rejects the token despite valid GitHub auth (the D-2/D-4 question)
**What goes wrong:** GitHub issues a token, but `GET /api/v1/tunnels` returns 401 because the relay only trusts tokens minted by its own registered GitHub App.
**Why it happens:** Dev Tunnels validates the token's issuing app. A `gh`-CLI token authenticates to *GitHub* but may be rejected by the *Dev Tunnels relay*.
**How to avoid:** The spike tests BOTH client IDs (`Iv1.e7b89e013f801f03` and `178c6fc778ccc68e1d6a`) against `list_tunnels`. The result settles D-4 (whether the agent's client ID must also bump).
**Warning signs:** Device flow completes, `list_tunnels` returns 401 not 200.

### Pitfall 4: `expires_in` absent Ôćĺ wrong expiry math
**What goes wrong:** For a non-expiring GitHub App token, the token endpoint omits `expires_in`; the Microsoft driver's `.unwrap_or(3600)` would mark a never-expiring token as expired in 1 hour, triggering pointless refresh attempts (which fail because there's no refresh token).
**How to avoid:** Treat absent `expires_in` as "no expiry" (store a far-future/sentinel `expires_at`, skip refresh when `refresh_token` is `None`). Confirm the actual case in the spike.

### Pitfall 5: Forgetting Phase 9.1 just moved these symbols TO Microsoft
**What goes wrong:** Treating this as a from-scratch GitHub build when 9.1 (commits 5ffa51e, 622d3fe) literally renamed GitHubÔćĺMicrosoft across menu/modal/footer. This phase reverses most of that cosmetic direction.
**How to avoid:** Read the 09.1-03/04 plans and SUMMARY files; this phase is largely `git`-level inverse of those edits plus the real driver swap.

## The Client-ID Spike (Plan 09.2-01) Design

**Goal:** Settle D-2/D-4 before any driver code. Two questions: (a) does device flow against `Iv1.e7b89e013f801f03` complete? (b) is the resulting token accepted by `GET /api/v1/tunnels` (200, not 401)?

**Recommended form: a single `#[ignore]`-gated integration test** in `crates/vector-tunnels/tests/github_devtunnel_spike.rs`, env-driven, run manually. Rationale over an xtask subcommand: reuses the existing `reqwest` + test harness, lives beside `live_devtunnel_smoke.rs` (same `#[ignore]` + env-gate convention the repo already uses, `REQUIREMENTS.md:75`), and leaves a permanent regression artifact. An xtask adds a new binary surface for a throwaway probe ÔÇö heavier.

**What the test does:**
1. For `client_id` in `[Iv1.e7b89e013f801f03, 178c6fc778ccc68e1d6a]`:
   - `POST https://github.com/login/device/code` with `Accept: application/json`, `client_id`. **Assert it does NOT return `device_flow_disabled`.** Print `user_code` + `verification_uri`.
   - Pause for the human to authorize (read `VECTOR_SPIKE_AUTHORIZE=1` or block on stdin / a generous sleep loop polling the token endpoint).
   - Poll `https://github.com/login/oauth/access_token` until `access_token` arrives. Record whether `refresh_token` and `expires_in` are present.
   - `GET https://global.rel.tunnels.api.visualstudio.com/api/v1/tunnels` with header `Authorization: github <token>`. **Assert 200**, print status + whether the body parses as a tunnel list.
2. Print a verdict table: per client ID Ôćĺ {device_flow_enabled?, refresh_token issued?, expires_in?, list_tunnels status}.

**Success:** `Iv1.e7b89e013f801f03` Ôćĺ device flow completes, `list_tunnels` 200. Ôćĺ Proceed with D-2 as written; agent unchanged (D-4 default).
**Partial:** `Iv1...` device flow completes but `list_tunnels` 401; `178c...` token gets 200. Ôćĺ Unlikely (gh-CLI app is not the relay's app) but if so, re-evaluate D-2.
**Failure mode A:** `Iv1...` returns `device_flow_disabled`. Ôćĺ Fall back per CONTEXT: loopback OAuth redirect with the GitHub App, or escalate. The agent's `gh`-CLI path becomes the comparison point.
