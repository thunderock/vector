# Phase 09.2 Plan 02: GitHub Device-Flow Driver + Token Store Summary

**A `GitHubAuth` RFC 8628 device-flow driver and `GitHubTokenStore` for the Dev Tunnels GitHub App (`Iv1.e7b89e013f801f03`): both POSTs set `Accept: application/json`, `slow_down` adds 5s (not doubled), `device_flow_disabled` is a typed fatal error, and the store persists a LIVE refresh token + real 8h expiry under `GITHUB_REFRESH_ACCOUNT`.**

## Refresh Path: LIVE

Per the 09.2-01 spike verdict, the Dev Tunnels GitHub App issues `refresh_token` + `expires_in=28800s (8h)`. Therefore:
- `GitHubTokenStore` persists the real refresh token and real expiry (no far-future sentinel on the live path).
- `GitHubAuth::refresh` is a live, exercised path (mapping `bad_refresh_token` -> `RefreshExpired`).
- The far-future sentinel (`NO_EXPIRY_SENTINEL_SECS`, ~100y) only applies if a token ever arrives WITHOUT `expires_in` — a safety net, not the live behavior. (Pitfall 4: NOT `unwrap_or(3600)`.)

## GitHub-Specific Divergences from the generic OAuth driver

1. **`Accept: application/json`** on both the device-code POST and the token POST. GitHub otherwise returns form-encoded bodies, which the JSON parser would reject.
2. **`slow_down` ADDS 5s** to the poll interval (`current_interval + 5s`, capped at 60s) — it does NOT double. Asserted in `polling_slow_down_adds_five_seconds` (>=5s and <8s elapsed).
3. **`device_flow_disabled`** is surfaced as `GitHubAuthError::DeviceFlowDisabled` from both `start_device_flow` and `poll_until_authorized`.
4. **`verification_uri` OR `verification_url`** fallback on parse.
5. **No scope param** (GitHub Apps ignore OAuth scopes).
6. **`bad_refresh_token`** (GitHub's name) -> `RefreshExpired` (Microsoft used `invalid_grant`).
